# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`ChimengSoso.github.io` — a GitHub Pages user site (deploys from the repo root, no `base` path needed), built entirely with Astro. It has two page groups sharing one site-wide navigation experience via Astro's `<ClientRouter />` (View Transitions):
- The home page (`src/pages/index.astro`) — a personal profile/link-hub with an interactive canvas starfield background (`src/components/StarfieldBackground.astro`).
- A Thai-language "knowledge" article section under `/knowledge` (`src/layouts/Layout.astro` + `src/pages/knowledge/*.astro`).

`public/` still holds static assets copied verbatim into the build output (`img.jpg`, `favicon.svg`, `promptpay-qr.png`, `robots.txt`, `knowledge/*` images) — but no longer any HTML/CSS of its own; there is no legacy static page bypassing the Astro build.

The planned feature backlog lives in **`docs/ROADMAP.md`** — when asked to "do the next feature" (or similar), pick from there, follow its per-item steps and verify/dependency rules, and tick its checkbox (+ note the commit hash) when done.

## Commands

- `npm run dev` — start Astro dev server (default port 4321)
- `npm run build` — build static site to `dist/`
- `npm run preview` — preview the production build locally
- `npm run check` — run `astro check` (full TypeScript type-check across `.astro`/`.ts`, incl. `<script>` blocks) — this is what catches the type errors the editor shows but `astro build` doesn't
- `npm run lint` — run ESLint (flat config in `eslint.config.js`: `typescript-eslint` + `eslint-plugin-astro`; `eslint . --fix` autofixes)

There is no test suite. **`astro build` does NOT type-check** — it compiles and can ship type-broken code. `npm run check` is the type gate and `npm run lint` the style/`any` gate, but **neither is wired into the build or the deploy workflow** (`.github/workflows/deploy.yml` only runs `astro build`), so they won't fail a deploy — run both manually before merging to `master`. After a non-trivial TS change, run `npm run check`: it surfaces the same errors the editor (WebStorm) shows, all at once, instead of finding them one screenshot at a time. `tsconfig.json` extends `astro/tsconfigs/strict` and excludes `eslint.config.js` (no point type-checking the lint config against the app config).

## Branching & deployment

- `master` is the default/production branch (**not** `main`) — only `master` triggers deployment. Do routine work on `develop` and merge into `master` when ready to publish; this is intentional, so pushes to `develop` don't go live.
- Pushing to `master` triggers `.github/workflows/deploy.yml`, which runs `withastro/action@v6` (installs deps, runs `astro build`) and deploys `dist/` to GitHub Pages via `actions/deploy-pages@v5`. There is no separate staging environment — every push to `master` goes live immediately.
- The repo's GitHub Pages source is set to **"GitHub Actions"** (not the legacy branch-based Pages source), so deployment only happens through this workflow. If the workflow's `on.push.branches` trigger is ever changed, remember `master` is the branch that matters here, not `main`.

## Commit messages

This repo's history is strictly single-line, imperative-mood, no body (e.g. `Add interactive starfield background and handle scramble effect`, `Fix profile page bugs and accessibility issues`). Don't add a bullet-point body or a `Why:` explanation paragraph — match the existing terse style. Only commit when the user explicitly asks (`commit ได้เลย` or similar); don't commit proactively after finishing a task.

## Adding a new article

Always touch two places:
1. Create the new `.astro` file in `src/pages/knowledge/`, using `<Layout>` the same way `claude-intro.astro` does. In the frontmatter: call `getArticle(href)` from `src/data/articles.ts` and `formatThaiDate` from `src/lib/dates`, derive `published`/`updated` strings, then pass them to `<ArticleByline>` (Layout no longer takes an `updated` prop — the byline renders the date itself) — no hand-written Thai date strings in the page file. Pass `headline={article.title}` so the `BlogPosting` JSON-LD headline stays suffix-free. Pass `prev`/`next` from `getAdjacentArticles(href)`. If the title contains slash-command tokens (e.g. `/grill-me`), wrap them in `<code>` in **both** the breadcrumb and the `<h1>` so they render as chips (see the slash-command-chip bullet in Architecture notes).
2. Add its entry in the `articles` array in `src/data/articles.ts` (typed `Article[]` — a missing/mistyped field now fails type-check in-editor), setting `soon: false`, `href` pointing at the new file in directory-style form (e.g. `my-article/`, not `my-article.html` — see the directory-style build note below), plus `dateISO` (YYYY-MM-DD publish date, e.g. `"2026-07-09"`) and `readingMinutes` (integer estimate — the listing shows both). Set `updatedISO` only when the article is later revised. Order in the array determines prev/next link order and reading order on the listing page — insert it where it should sit in the sequence. To estimate `readingMinutes`, count the Thai characters in the article body (a decent prose proxy — strip tags/`<style>`/`<script>`/code) and divide by ~1000–1100; it's shown as "อ่าน N นาที" and is easy to tweak by hand later.

The article's Open Graph share image is generated automatically — do **not** set an `image` prop or add a screenshot asset for it. `Layout.astro` derives `og:image` from the page URL and `src/pages/og/[slug].png.ts` renders a branded card from the entry's `title`/`tag` (see the Open Graph bullet in Architecture notes).

## Architecture notes

- **`astro.config.mjs`** sets `site` but no `base`, and relies on Astro's default `build.format: "directory"` output (e.g. `src/pages/knowledge/claude-intro.astro` → `/knowledge/claude-intro/index.html`; `src/pages/index.astro` still emits a flat `dist/index.html` at the root, same as before). GitHub Pages serves directory indexes natively, so links use `href="claude-intro/"` style paths, not `.html` suffixes.
- **`src/pages/index.astro`** is a standalone page — it does *not* wrap in `Layout.astro`, since the profile card's centered layout is fundamentally different from the two-column reading layout `Layout.astro` provides (no TOC, no donate button, no site footer). It renders its own `<ClientRouter />` and `<StarfieldBackground />` independently of `Layout.astro`, which does the same for every `/knowledge/*` page — both must keep doing so for View Transitions and the persisted canvas to work site-wide.
- **`src/components/StarfieldBackground.astro`** renders the `<canvas>` background shared by the home page and every knowledge page. It uses `transition:persist` so the same DOM node (and its running particle simulation) survives client-side navigation instead of being torn down and rebuilt: leaving `/` for a knowledge article scatters the stars outward and fades/pauses the canvas; returning reforms and fades it back in. All mutable animation state lives on the canvas node itself (`canvas.__starfield`), not in script closures, because closures aren't guaranteed to survive Astro's script re-execution on each swap — see the component for the `astro:page-load` + `data-bg` (`"interactive"` on `/`, `"plain"` on `Layout.astro`) lifecycle.
- **Emerald-star catch mini-game + clue hunt (in `StarfieldBackground.astro`).** A rare emerald star drifts across the upper sky on `/`; clicking it scores a catch, 6 catches = one "round"/trophy, difficulty ramps within a round (`SPEED_MAX_FACTOR`). All tunables are constants grouped at the top of the `<script>` (`GOLD_*`, `STAR_COLOR`, `ROUND_TARGET`, `TRAIL_MAX`, `ROUND_CELEBRATE_MS`, `CLUES`, …). **Two separate counters, deliberately different lifetimes:** `game.total` (session score shown in the HUD) is **in-memory only** — a full refresh resets it (chosen to reduce the hack surface, accepting lower UX). `progress` (completed-round count, 0..`PROGRESS_MAX`) **is persisted** via `localStorage` key `starfieldProgress` (load/save in the shared `src/lib/starClues.ts`); it drives the whispered clue ladder that teaches the hidden SecretConsole puzzle (Konami → `chi` username → anagram password), the re-openable "codex" panel (opened by the HUD trophy button), and the tier-10 worked-example reward. Neither counter is real security: editing `starfieldProgress` in storage only spoils your own clue hunt, and `/divine-lore/` still verifies its own token (`divineLoreToken.ts`) independently — so revealing clues never bypasses the actual gate. `SecretConsole.astro` reads `progress` (via `starClues.ts`) to add a gatekeeper acknowledgment line on `ACCESS GRANTED` for players who followed the stars. Any new interactive overlay here (the codex, like `.profile`/`.secret-console-overlay` before it) must be added to the black-hole `pointerdown` exclusion list.
- **`src/layouts/Layout.astro`** is the shared shell for all `/knowledge/*` pages (blue color theme, Sarabun font with fluid `clamp()`-based sizing controlled by the single `--scale` CSS variable, card/tag/breadcrumb/callout/references styling, a sticky "on this page" TOC auto-built from each page's `h2`s). New Astro content pages should use this layout via `<Layout title=... description=...>` rather than duplicating markup — see `src/pages/knowledge/claude-intro.astro` for the pattern. Layout also renders the prev/next article nav and the "last updated" footer itself, driven by the `prev`/`next`/`updated` props — don't hand-write a `<footer>` in article pages. The top-of-article header row (`.article-head`, holding the breadcrumb + `<ArticleByline>` publish/updated date at top-right) uses **CSS Grid**, not flex: `grid-template-columns:minmax(0, max-content) 1fr` (≥480px; single column below). The breadcrumb column prefers its one-line width but can shrink/wrap as a last resort; the date column (`1fr` + `text-align:right`) pins the date to the right edge and wraps it to two lines when space is tight (`ArticleByline` keeps each date `nowrap` and hides the " · " separator on wrap via a small script). **Don't try to make the breadcrumb never-wrap** — that reintroduces a long saga of the date overflowing the card / breadcrumb words dropping; a rare trailing-word wrap at extreme zoom is the deliberate, accepted tradeoff.
- **`src/data/articles.ts`** (typed `Article[]`) is the single source of truth for the knowledge article list (used by the listing page) and reading order (used by `getAdjacentArticles(href)` for prev/next links). `soon: true/false` marks placeholder ("เร็วๆ นี้") vs. published entries; only published entries count for prev/next. Each published entry also carries `dateISO` (YYYY-MM-DD publish date), optional `updatedISO` (YYYY-MM-DD), and `readingMinutes` (all optional so `soon` placeholders can omit them). Thai display strings are derived via `formatThaiDate`/`formatThaiDateShort` in `src/lib/dates.ts` — never store hand-written Thai date strings in the array. `getArticle(href)` looks up a single entry by href.
- **The `/knowledge` listing (`src/pages/knowledge/index.astro`)** groups cards by `tag` (preserving array order, so first-seen tag = section order: intro → Claude Code → มุมมอง) under a category header with a count, instead of repeating the tag pill on every card. Each card shows the title, desc, and a meta row (`อ่าน N นาที` + short date via `formatThaiDateShort(a.dateISO)` from `src/lib/dates.ts`) plus a hover "อ่านต่อ" chevron. Design rationale is research-backed (scannability / F-pattern / visual hierarchy). Slash-command tokens in `title`/`desc` render as code chips via the in-file `splitCommands`/`isCommand` helpers (see the slash-chip bullet below).
- **Slash-command tokens render as inline-code chips everywhere a title/description appears:** the article `<h1>` and breadcrumb (hand-wrapped in `<code>` in the page file), the `/knowledge` listing card title+desc (auto via `splitCommands`/`isCommand`), and the OG card (auto-detected in `card.ts`). Styling lives with each context — `.breadcrumb code` / `h1 code` in `Layout.astro`'s global `<style>`, `.item h3 code`/`.item p code` scoped in the listing. The breadcrumb chip also opts out of the breadcrumb's `text-transform:uppercase`/`letter-spacing` so the command reads literally (`/grill-me`, not `/GRILL-ME`). Facebook's share title is plain `<title>` text and can't be chip-formatted — that's expected, leave it.
- **Open Graph share cards are generated at build** by `src/pages/og/[slug].png.ts` (endpoint) + `src/og/card.ts` (renderer). Every `/knowledge/*` page's `og:image` is auto-derived in `Layout.astro` from the pathname (`/og/<slug>.png`); the endpoint renders one 1200×630 branded PNG per published article (read from `articles.ts`) plus one for the `/knowledge` listing, showing the article `title` + `tag` on the site's blue theme. Slash-command tokens in the title (words starting with `/`, e.g. `/grill-me`) are auto-detected and drawn as inline-code chips in JetBrains Mono. Fonts (`Sarabun-Bold.ttf`, `JetBrainsMono-Bold.ttf`) live in `src/og/` and are read at build (see the Thai-font gotcha below for why the rendering is set up the way it is). The `fb:app_id` meta tag in `Layout.astro` and `index.astro` is a **public** value (it appears in every page's HTML, so it needs no protection) — only a Facebook App *Secret* would be sensitive, and none is used here.
- **`src/pages/privacy.astro`** is the bilingual (TH/EN toggle via `data-lang` + `localStorage.privacyLang`) privacy policy, linked from the Layout site footer and the home page's fixed bottom-left `.privacy-link`. **Maintenance rule: any change to what the site stores client-side (new `localStorage`/`sessionStorage` key) or any new third-party request must be reflected in BOTH language blocks, and bump the "last updated" date.** Dates site-wide use CE years ("2026"), never พ.ศ. — a 2569 slipped in once. `src/pages/404.astro` is the branded not-found page (Astro emits `dist/404.html`, which GitHub Pages serves automatically for unknown URLs).
- **`src/data/socialLinks.ts`** (typed `SocialLink[]`) is the single source of truth for social profile URLs + their inline-SVG icon path data, consumed by `index.astro` (profile icons and the JSON-LD `sameAs` list) and `Layout.astro` (site footer, via the filtered `footerSocialLinks` export). All icons site-wide are inline SVGs — the Font Awesome CDN stylesheet was deliberately removed (it was render-blocking external CSS on every knowledge page); don't reintroduce an icon font for new icons.
- **`src/styles/fonts.css`** — self-hosted `@fontsource` imports (replaced the old Google Fonts `<link>`s). **Critical invariant: both `index.astro` and `Layout.astro` must import this same file**, so all `@font-face` rules land in one shared CSS chunk referenced by both page groups (verified in dist: one shared chunk with all 16 rules). This is the self-hosted successor to the old "same Google-Fonts URL on purpose" trick — same rationale: a single shared stylesheet means ClientRouter's `/` ↔ `/knowledge/*` swap never blocks on loading a new stylesheet. Intentional subsets: Sarabun latin+thai (body), Chakra Petch latin+thai (the mini-game HUD/clues/codex are Thai — don't drop the thai subset), Fira Code + JetBrains Mono latin-only (those families have **no** Thai subset; latin-only is correct there). Don't add new `@fontsource` packages without asking (see Conventions).
- **`src/pages/rss.xml.ts`** — RSS endpoint fed by `articles.ts` (published entries only, directory-style permalinks). The `<link rel="alternate" type="application/rss+xml" href="/rss.xml">` lives in both `Layout.astro` and `index.astro` heads. Declares `xmlns:atom` and emits an `<atom:link rel="self">` (via `customData`) so it passes W3C feed validation cleanly.
- **Cloudflare Web Analytics beacon** — a `<script is:inline … data-cf-beacon>` in both `Layout.astro` and `index.astro` heads. The token is a **public** value (like `fb:app_id`; it ships in every page's HTML, so needs no protection). `"spa": true` makes it count ClientRouter's pushState navigations, not just hard loads. Disclosed in `privacy.astro` (both TH+EN). The beacon is **skipped on `/divine-lore/`** (gated by `analyticsEnabled = segments[0] !== "divine-lore"` in `Layout.astro`) so the hidden page stays out of the dashboard — safe because that page is only reached via SecretConsole's hard `window.location` redirect (a full load), never a client-side swap.
- **`Layout.astro`'s `BlogPosting` JSON-LD** (article pages only, gated by `isArticle = type === "article" && !!dateISO`) sets `headline` from the optional `headline` prop (falling back to `title`) — article pages pass `headline={article.title}` so the JSON-LD headline is the clean title without the ` | หมวดความรู้` suffix. The JSON-LD `image` reuses the same `ogImageUrl` computed once above (don't re-derive the OG path). `ShareButtons` is likewise gated by `isArticle`, not by `updated` (so articles without an `updatedISO` still show the share row, and `privacy.astro` — `type="article"` but no `dateISO` — stays share-free).
- **`index.astro`'s scramble word lists** (handle aliases, role titles) are defined once in the frontmatter and rendered into both the `aria-label`s and `data-scramble-words` JSON attributes; the scramble script reads the words back from those attributes, so there is no second copy to keep in sync inside the `<script>` block.

## Conventions

- **Reduced motion**: any new CSS/JS animation needs a `@media (prefers-reduced-motion: reduce)` fallback (instant state change, no transition) — see `profile.css` and `StarfieldBackground.astro`.
- **Hover vs. touch branching**: use `@media (hover: hover)` / `@media (hover: none)` to detect whether the primary input can hover, not touch-event feature detection (`ontouchstart` etc.) — handles hybrid touchscreen laptops correctly. Existing usages: `DonateButton.astro`, `profile.css`.
- **Accessible show/hide**: elements that must stay reachable by keyboard/tab order and screen readers (e.g. `.featured-link`) are hidden via `opacity`/`max-height`/`pointer-events`, never `display: none` or `visibility: hidden`. `display: none` is still fine for purely decorative icon-swap toggles (e.g. copy/check icon states in `CodeBlock.astro`/`DonateButton.astro`/`ShareButtons.astro`) where nothing needs to stay perceivable either way.
- **Never use `any` — anywhere, and never silently.** This is a hard rule across the whole codebase (`.ts`, `.astro` frontmatter, and `<script>` blocks), not just script blocks. `@typescript-eslint/no-explicit-any` is on, and `tsconfig`'s strict `noImplicitAny` flags untyped function params in `<script>` blocks (the language server surfaces these even though `astro build`/`eslint` won't fail on implicit any). If you think a value genuinely needs `any`, **stop and ask the user first** — do not reach for it on your own. The typed alternatives, in order of preference: give the value a real type/interface; use `unknown` + a narrowing guard (e.g. `JSON.parse` returns `any`, so assign to `unknown` then validate with `Array.isArray` / a `w is string` type-predicate before use — see `index.astro`'s scramble init); or, when a value needs a custom property absent from its DOM/global type (`window.__flag`, `canvas.__starfield`), define a named intersection type (`type Foo = Window & { __flag?: boolean }`) and cast once into a local, never scattering `(x as any).prop`. Event handlers get typed params (`(e: MouseEvent)`), and narrow `EventTarget` via `e.target instanceof Element` rather than casting to `any`. See `StarfieldBackground.astro`'s `StarfieldCanvas`, `SelfXssWarning.astro`'s `WindowWithSelfXssFlag`, and `index.astro`'s fully-typed scramble/tilt scripts.
- **Don't add new npm dependencies without asking first**, even for tooling (e.g. `astro check` wants to install `@astrojs/check`/`typescript`) — confirm with the user before changing the dependency footprint.
- **UI changes get verified in a live browser** (via the preview tooling), not just `npm run build` — a clean build only proves the code compiles, not that the feature behaves correctly across navigations/interactions.
- **Verify cadence is the user's call — ask once per session, then hold to it.** Two modes: (a) *incremental* — run `npm run check` + `npm run lint` + browser verify after each change, or (b) *batch* — just edit and let the user eyeball it, deferring all compile/type/lint/browser checks to a single pass right before commit (to save tokens/time). At the start of a session where this matters, ask which they want; once they pick, follow that mode for the rest of the session without re-asking. Only revisit if the user changes their mind. A new session resets this — ask again, don't assume last session's choice.

## Gotchas learned the hard way

- **`<script>` tags stop running after client-side navigation unless wired to `astro:page-load`.** With `<ClientRouter />` site-wide, top-level script code only ever executes once (ES module semantics) — after the router swaps in new DOM, any code that ran at the top level is now bound to detached elements. Wrap all init logic (querySelector, addEventListener, `new IntersectionObserver`, etc.) in `document.addEventListener('astro:page-load', () => { ... })` instead of running it bare, in *every* component/page with a script, not just ones that feel "SPA-relevant." This bit `Layout.astro`'s TOC builder, `DonateButton`, `ShareButtons`, `CodeBlock`, and even `index.astro`'s own tilt/reveal/handle-scramble scripts — missed on the first pass and only caught by manually navigating back and forth in a live browser, not by `astro build`.
  - Corollary: if you create a new `IntersectionObserver` (or similar) inside an `astro:page-load` handler, `.disconnect()` the previous one first — otherwise every navigation back to that page leaks another observer watching stale/detached nodes.
- **State for a `transition:persist`-ed element must live on the DOM node itself, not in script closures.** Even though the node survives navigation, the inline `<script>`'s module-level variables are not guaranteed to. Pattern used here: `canvas.__starfield = {...}` holds all mutable animation state, and `astro:page-load` checks `canvas.__starfield` to decide "fresh boot" vs. "resume."
- **When refactoring a function that ends in `return { ... }` into one that needs to derive extra properties (e.g. via `Object.assign`), actually convert the `return` into a `const x = { ... }` assignment.** Once, mid-refactor, the original `return { ... };` was left in place after adding an `Object.assign(...)` line after it — the new code became unreachable dead code with zero console errors (nothing threw, it just silently never ran). `astro build` and TypeScript didn't catch it either, since it's valid-but-dead code. Caught only via live browser testing that asserted on actual runtime state (e.g. checking a property existed on the object) rather than just "no errors in the console." Lesson: after any non-trivial refactor of stateful init code, verify the *new* behavior actually executes — absence of errors is not evidence of correctness for dead code.
- **Any new interactive overlay/UI on `/` must be excluded from StarfieldBackground's black-hole `pointerdown` handler and from profile.css's body-wide `user-select: none`.** The handler calls `e.preventDefault()` on every pointerdown outside `.profile` (to stop the drag from starting a native selection), which silently breaks placing the text caret with the mouse in any input rendered over it — no error, clicks just don't move the caret (bit SecretConsole's username/password fields). Fix both sides: add the overlay's class to the black-hole `pointerdown` exclusion list in `StarfieldBackground.astro` (the `closest('.profile, .secret-console-overlay, .gold-total, .gold-codex, .privacy-link')` call — grow that list), and set `user-select: text` on text inputs inside it. Even a plain link needs the exclusion: pointerdown outside it engages the black hole, so clicking would detonate a shockwave under the element. Related: toggling an `<input>`'s `type` (password↔text) resets its caret to the start — capture `selectionStart/End` before the switch and `setSelectionRange` after, and `preventDefault()` the toggle button's `mousedown` so it doesn't steal focus.
- **`preview_screenshot` times out (~30s) on this Windows box even when the page is perfectly healthy** — evals respond instantly, console is clean, and pausing the starfield rAF loop does not help; it's the capture tool, not the site. Don't retry more than once and don't treat it as a page bug. Verify with `preview_eval` (runtime state), `preview_inspect` (computed styles), and `preview_snapshot` (structure/text) instead — they cover everything a screenshot would and more precisely.
- **Driving pointer-based UI from `preview_eval` needs two workarounds.** (1) Synthetic `PointerEvent`s carry a `pointerId` that doesn't exist, so any handler calling `setPointerCapture` throws `NotFoundError` and silently aborts mid-handler (bit the scrubber test) — stub it first: `el.setPointerCapture = () => {}`. (2) `preview_click` on a link + an immediate `preview_eval` races the ClientRouter navigation — the eval can read the *old* page's `location`/DOM; re-check in a follow-up eval (or `element.click()` in one eval, assert in the next) before concluding navigation failed.
- **JSX comments `{/* ... */}` can't be a sibling of the single element returned from a `.map()` arrow** (`items.map((a) => ( {/* boom */} <a>...</a> ))`) — `astro check` fails with a cryptic cascade of `ts(1128) Declaration or statement expected` pointing at the end of the file, nowhere near the comment. Put template comments where children are allowed (e.g. just inside the parent container, above the `{items.map(...)}` expression).
- **When editing Thai copy, check for smuggled zero-width spaces before committing** — a U+200B slipped into a privacy-page edit once and is invisible in every editor. `Grep pattern '\x{200B}'` over the touched files catches it.
- **Verifying rAF-driven canvas animation in a preview tab: don't wait for real frames — step the simulation manually.** Browsers throttle `requestAnimationFrame` in backgrounded/preview tabs, so "reload, wait N seconds, screenshot" is flaky and can assert against a stale frame. The starfield exposes its full state (including `tickFrame(t)`) on the canvas node as `canvas.__starfield`; drive it deterministically from `preview_eval` by calling `tickFrame` with synthetic timestamps, then assert on runtime state (particle positions/velocities/flags) or wrap the 2D-context draw methods with a spy to check what was actually drawn. Screenshots are for final visual confirmation only, not for verifying behavior.
- **Ambient particle drift must be a smooth arc, not per-frame random jitter.** Randomly re-aiming a particle's velocity every frame reads as buzzing insects/fireflies, not celestial bodies. The pattern that looks right: give each particle a constant slow speed and a small fixed per-frame turn rate so its path curves gently (see `DRIFT_SPEED_MIN`/`DRIFT_SPEED_RANGE`/`DRIFT_TURN_MAX` in `StarfieldBackground.astro`). Randomness belongs in the initial parameters (speed, heading, turn direction), not in the per-frame update.
- **Astro's scoped `<style>` blocks silently don't apply to elements created at runtime via `document.createElement`.** Astro scopes CSS by tagging template elements with a `data-astro-cid-*` attribute and rewriting selectors to require it; a `<div>` built and appended by a `<script>` (e.g. a log line in `SecretConsole.astro`) never gets that attribute, so a rule like `.log .log-denied { color: red }` compiles to effectively require the attribute on `.log-denied` too and never matches — no build error, no console error, the text is just always the default color. Fix: wrap the dynamic class in `:global(...)`, e.g. `.log :global(.log-denied) { color: red }`. Applies to any component that builds DOM nodes in JS and expects a scoped `<style>` block to color/style them.
- **JSX can't live in Astro frontmatter (the `---` fence) — only in the template below it.** A helper that *returns markup* (e.g. `const withChips = (t) => t.split(re).map((s) => <code>{s}</code>)`) defined in the frontmatter fails the build with a cryptic `Unterminated regular expression` (the compiler tries to parse `<code>` as TS and chokes on the following `/`), and `astro check` may not flag it. Keep frontmatter helpers pure data/logic (e.g. `splitCommands` returns a `string[]`, `isCommand` returns a `boolean`) and do the `<code>` wrapping inline in the template: `{splitCommands(a.title).map((seg) => isCommand(seg) ? <code>{seg}</code> : seg)}`. This also keeps scoped `<style>` working, since the `<code>` is authored in the component's own template. Bonus gotcha it caused: a build-only error invisible to `astro check` — always `npm run build`, not just `check`, after touching frontmatter that borders on JSX. `src/og/card.ts` rasterises an SVG of Thai+Latin text. Three approaches were tried and only the third works everywhere:
  1. sharp/librsvg with the font embedded as a base64 `@font-face` renders fine on the Windows dev box, but the **Linux CI's librsvg ignores the embedded font** — Thai comes out as `.notdef` boxes (Latin falls back to a system serif; Thai has no fallback and shows Unicode codepoint boxes). `astro build` is completely silent about it, so it only surfaces on the deployed site.
  2. `@resvg/resvg-js` loads fonts deterministically (explicit `fontBuffers`, `loadSystemFonts:false`) but its shaper **drops Thai mark-to-mark positioning** — tone marks stacked over an upper vowel (e.g. ไม้เอก over สระอี in "ที่") collapse onto the vowel and become invisible.
  3. **The working combo:** sharp/librsvg (its pango+harfbuzz stack shapes Thai correctly) + a generated **fontconfig** config. `card.ts` writes a `fonts.conf` pointing `<dir>` at `src/og/` and sets `FONTCONFIG_FILE` **before sharp is imported** (sharp is imported lazily inside `renderCard` for this reason), with a `<match>` that force-assigns every non-`JetBrains Mono` family to `Sarabun` so no system font can leak in for Latin — deterministic *and* correctly shaped.
